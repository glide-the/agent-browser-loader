两个插件，职责拆分：

## 1. agent-browser-plugin-userprofile-browser（browser.provider）

作为浏览器 provider（`--provider userprofile-browser`）。一次性把真实 Chrome Profile
全量 rsync 到独立的 `RemoteDebug` user-data-dir，绕过 Chrome “non-default data directory”
检测与运行中 Profile 锁，然后直接用调试端口拉起 Chrome 并返回 CDP URL，由 agent-browser
接管驱动：

- `browser.launch`：rsync 同步（状态文件存在则跳过，`force:true` 重新同步）→ spawn Chrome（`--remote-debugging-port`、`--user-data-dir=<debug>`、`--profile-directory`）→ 从 `DevToolsActivePort` 解析 cdpUrl → 返回 `{ browser: { cdpUrl, directPage, metadata, cleanup:{ sessionId } } }`；若提供 `cdpUrl` 则走 connect 模式直接返回。
- `browser.close`：按 `sessionId` 从会话注册表查到 Chrome 并终止（`removeDebugDir:true` 时一并删除 debug 目录），返回 `{ data: { closed } }`。

## 2. agent-browser-plugin-stealth（launch.mutate）

本地 launch 前注入 stealth 的 args / extensions / initScripts / userAgent，
并从状态文件读取 Profile 参数（`--user-data-dir`、`--profile-directory`），
从而可继续使用 `--extension`。流程：
插件进程被启动 → stdin 收到 JSON 请求 → 返回 launch（args/extensions/initScripts/userAgent）。

## 配置改为本地文件（不再依赖环境变量）

agent-browser 以子进程方式拉起插件，环境变量无法可靠传入，因此 Profile 相关配置
改为读取本地文件 `<cwd>/.agent-browser/userprofile.config.json`（两个插件共享）：

```json
{ "userDataDir": "源 Chrome 目录", "profileDirectory": "Default", "debugDir": "RemoteDebug 目录", "statePath": "状态文件路径" }
```

字段均可选。`AGENT_BROWSER_USERPROFILE_*` 等环境变量仅作为兜底，配置文件为主。
