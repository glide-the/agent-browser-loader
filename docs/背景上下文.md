和 agent-browser 的关系

- 这个目录里的实现是浏览器扩展模式，不是 `agent-browser.plugin.v1` 的 stdin/stdout 进程插件。
- 如果你要在 agent-browser 中使用它，通常走“浏览器扩展加载”路径，而不是 agent-browser plugin add ... 路径。

**步骤一：使用命令行启动您的日常 Chrome**

在启动时加上 `--remote-debugging-port=9222` 参数，暴露远程控制接口。

- **Windows:**在终端中运行（需包含 Chrome 的安装路径）：`"C:\Program Files\Google\Chrome\Application\chrome.exe" --remote-debugging-port=9222`
- **macOS:**在终端中运行：`"/Applications/Google Chrome.app/Contents/MacOS/Google Chrome" --remote-debugging-port=9222`
- **Linux:**在终端中运行：`google-chrome --remote-debugging-port=9222`

**步骤二：让 Agent Browser 连接到该浏览器**

启动日常浏览器后，在您的 agent-browser 项目或 CLI 终端中，使用以下方式进行连接：

**方法 1：使用自动连接（最推荐）**使用 `--auto-connect` 参数启动 `agent-browser`。它会自动检测本地正在运行且开启了 9222 端口的 Chrome。 [1, 2]

**bash**

`agent-browser --auto-connect`

请谨慎使用此类代码。

**方法 2：指定 CDP 端口**如果遇到多开或特殊端口需求，可以通过 `--cdp` 参数明确指定连接端口： [1]

**bash**

`agent-browser --cdp 9222 open https://www.google.com`

请谨慎使用此类代码。

**方法 3：使用 API 模式（以 `browser-use` 为例）**如果您是在代码中使用 `agent-browser` 的 SDK，可以通过指定 `cdp_url` 来接管当前会话： [1, 2]

**python**

`from browser_use import Agent, Browser

# 连接到本地正在运行的浏览器
browser = Browser(cdp_url="http://127.0.0.1:9222")
agent = Agent(task="在此浏览器中执行任务", browser=browser)`

请谨慎使用此类代码。

发现boss直聘还有反爬虫策略

要隐藏或绕过这些检测，你需要利用 CDP 本身的能力在页面加载前注入反检测脚本（如 `stealth.js`） 并抹除相关特征。 [1]

以下是具体的解决步骤和方案：

**1. 注入反检测脚本 (Stealth Scripts)**

在自动化框架（如 Puppeteer、Playwright）中，可以通过 CDP 的 `Page.addScriptToEvaluateOnNewDocument` 方法在页面加载最开始时执行 JS 脚本，覆盖掉暴露身份的 DOM 属性。 [1, 2]

你可以在自动化工具中挂载常用的 **puppeteer-extra-plugin-stealth** 或参考其原理注入以下代码：

**javascript**

`// 抹除 navigator.webdriver 属性
Object.defineProperty(navigator, 'webdriver', {
  get: () => undefined
});

// 移除 CDP 注入的插件特征
delete navigator.plugins;
navigator.plugins = [1, 2, 3, 4, 5];

// 隐藏自动化工具特有的环境变量
window.navigator.chrome = {
  runtime: {}
};`

请谨慎使用此类代码。

**2. 覆盖 CDP 隐式参数**

某些高级反爬虫系统会检测浏览器是否以远程调试端口（`--remote-debugging-port`） 启动。直接连接真实用户的浏览器实例可以规避此问题。使用 Puppeteer 或 Playwright 时，可以配置启动参数（Arguments），让浏览器表现得与日常使用完全一致： [1, 2]

- 禁用自动化扩展标志 `-disable-blink-features=AutomationControlled`
- 使用真实的 User-Agent 字符串
- 避免使用指纹特征异常的“无头模式 (Headless)” [1]

**3. 规避前端代码对 DevTools 的检测**

如果网站试图通过检测窗口尺寸变化、DevTools 审查元素（如 `console.profile()` 或无线循环 debugger 语句）来判断开发者工具是否打开，可以在浏览器端通过脚本禁用控制台监听。 [1, 2, 3]

- **安装浏览器反检测扩展**：例如在 Chrome 使用特定的插件拦截 F12 和右键调试拦截。
- **使用代理工具重写响应**：利用 Charles 等工具将网站中检测 DevTools 的 JS 逻辑直接替换为空，防止触发前端风控。 [1, 2]

**4. 彻底掩盖 CDP 痕迹（高级进阶）**

对于严格的风控平台，如果标准的 stealth 脚本仍然被拦截，通常需要使用定制化的框架（如 Patchright 或 undetected-chromedriver），在 CDP 通信的最底层修补 ChromeDriver 特征，使每一次网络请求的指纹和真实的正常浏览器完全一致。 [1]

**Further Exploration**

- 了解在复杂的平台风控下如何结合真实浏览器进行反检测实战，请阅读 **Yousali Blog**。
- 探索如何在使用 Playwright 时处理常见自动化检测以及脚本注入，请参考 **GitHub CrawlerTutorial**。 [1, 2]
