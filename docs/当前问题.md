过程通过plugin 来做jsonp rpc通信，通过本地协议的方式进一步抽象了capabilities 概念

| **Capability  能力** | **Request type  请求类型** | **How users invoke it  用户如何调用** | **Response field  响应字段** |
| --- | --- | --- | --- |
| `credential.read` | `credential.resolve` | `auth login --credential-provider <name>` | `credential` |
| `browser.provider` | `browser.launch`, `browser.close` | `--provider <name>` | `browser` |
| `launch.mutate` | `launch.mutate` | Any local launch  任何本地启动 | `launch` |
| `command.run` | Custom request type  自定义请求类型 | `plugin run <name> <type>` | `data` |

当前结论：

- `agent-browser-plugin-userprofile-browser` 实现为 `browser.provider`（`--provider userprofile-browser`）：rsync 同步真实 Profile 到 RemoteDebug 目录后，直接用调试端口拉起 Chrome 并返回 CDP URL，由 agent-browser 接管驱动；`browser.launch` 响应字段为 `browser`，`browser.close` 按 `sessionId` 终止 Chrome 返回 `data`。
- 由 provider 自己启动 Chrome，因此扩展、Profile、stealth 参数都可由 provider 直接控制，规避了 `--provider` 与 `--extension` 不能同用的限制。
- `agent-browser-plugin-stealth`（`launch.mutate`）仍用于本地 launch 路径：从状态文件读取并追加 `--user-data-dir` / `--profile-directory`。
- **环境变量无法使用**：agent-browser 以子进程拉起插件，env 变量无法可靠传入。Profile 相关配置改为读取本地文件 `<cwd>/.agent-browser/userprofile.config.json`（两个插件共享，字段 `userDataDir`/`profileDirectory`/`debugDir`/`statePath`），`AGENT_BROWSER_USERPROFILE_*` 仅作兜底。
